# 一种基于 AI 大模型的本地优先文件整理与检索方式

## 题目
一种面向大学生用户的本地优先文件整理与检索助手：基于高性能索引、规则引擎与多模型协同的工程实现

## 摘要
面向不熟悉 AI Agent、Windows 文件管理流程及终端命令的大学生用户，本文提出并实现一种本地优先的文件整理与检索方法。该方法以 Everything 作为高性能本地索引引擎完成候选文件发现，以规则引擎和 AI 模型协同完成语义理解、排序规划与操作建议，并通过“可预览、可确认、可撤销”的执行机制降低批量操作风险。系统采用 Electron 构建图形交互层，Rust 实现检索与整理核心能力，在保证交互易用性的同时兼顾执行效率与本地可控性。项目当前已实现自然语言检索触发、排序计划生成与确认执行、变更记录输出等功能。本文给出系统设计、关键实现、可复现实验方案与扩展方向，为桌面学习环境中的个人文件治理提供可落地路径。

## 关键词
本地优先；文件整理；语义检索；Everything；AI Agent；人机协同

## 1 引言
在大学生日常学习中，桌面、下载目录与课程资料目录会随时间快速累积，导致以下问题。

1. 文件命名不规范，用户难以回忆路径与版本。
2. 手工归档成本高，跨课程和跨项目资料混放严重。
3. Windows 原生搜索在语义理解与意图驱动整理方面能力有限。

与此同时，纯云端 AI 助手虽然具备理解能力，但在隐私、成本与执行可控性上存在约束。基于此，本文选择本地优先路线：将高频文件发现与操作执行留在本地，将大模型用于规划与理解，从而在可用性、安全性与成本之间取得平衡。

## 2 问题定义与设计目标

### 2.1 问题定义
目标用户为对终端命令和复杂配置不熟悉的大学生。其核心任务是“用自然语言找到文件并完成可控整理”。系统需解决三类痛点。

1. 检索痛点：用户只记得用途，不记得文件名与路径。
2. 整理痛点：批量移动操作易误伤，缺少可验证流程。
3. 学习成本痛点：命令行门槛高，操作反馈不透明。

### 2.2 设计目标
1. 低门槛：图形界面 + 自然语言交互。
2. 高效率：本地索引优先，减少全盘扫描与模型无效读取。
3. 高安全：先规划后执行，用户确认后落地。
4. 可追溯：执行后输出变更记录，支持回看与恢复。
5. 可扩展：支持多模型接入与本地模型替换。

## 3 系统总体方案

### 3.1 总体架构
系统采用四层架构。

1. 交互层：Electron 双窗口界面（宠物窗口 + 聊天窗口），提供自然语言输入、附件输入与执行反馈。
2. 文件发现层：Rust 二进制调用 Everything HTTP 接口完成候选文件检索；网络不可用时回退本地遍历。
3. 语义与规划层：基于多模型协同策略生成检索解释与整理计划。
4. 执行与安全层：Rust 排序引擎执行受限操作（创建目录、移动文件），输出变更日志。

### 3.2 交互层方案
系统在桌面路径附近运行类 Claude Code 风格的 Agent 交互，支持自然语言任务分解。用户可以发起“查找、归类、整理、打开”类请求，系统先返回可理解的执行思路，再在用户确认后进行操作。对于未授权目录，系统通过附件目录白名单进行控制，避免默认读取全部磁盘内容。

### 3.3 文件发现层方案
Everything 负责索引与高速检索，优势来自 NTFS 元数据级查找。系统仅将其作为候选生成器，不直接执行复杂语义判断。候选结果交由后续模块做筛选、重排与解释，减少模型 Token 消耗和无关文件读取。

### 3.4 模型协同方案
系统支持轻量、标准、增强三档模型协作（可映射到 Haiku、Sonnet、Opus 思路）。

1. 轻量模型：快速判断任务类型与难度，处理低复杂度请求。
2. 标准模型：承担主流程理解、候选筛选与重排。
3. 增强模型：处理复杂规划、边界情况与多步骤任务。

模型接入方式为 Base URL + API Key 的云端服务，也支持通过本地部署渠道（如 Ollama）降低成本。

### 3.5 安全执行方案
整理任务采用两阶段机制。

1. 规划阶段：生成 JSON 计划并预览。
2. 执行阶段：仅在用户确认后应用计划。

执行边界默认不包含删除操作，只允许创建目录和移动文件，从机制上降低误删风险。

## 4 项目实现与代码映射

### 4.1 工程结构概览
本仓库为 Electron + Rust 混合工程。

1. Electron 主进程与渲染层位于 electron 目录。
2. Rust 检索、排序与模型桥接能力位于 src 目录。
3. 可执行入口位于 src/bin 目录。

### 4.2 核心模块对应关系
| 模块 | 主要职责 | 关键实现文件 |
|---|---|---|
| UI 与技能路由 | 聊天交互、技能触发、计划管理 | electron/main.js |
| Everything 检索 | HTTP 检索 + 本地回退检索 | src/CCEverything.rs, src/bin/cceverything.rs |
| 排序执行引擎 | 扫描、分类、计划、应用 | src/CCSort.rs, src/bin/ccsort.rs |
| 模型调用桥接 | 运行 Claude CLI、合并设置 | src/CCConnect.rs, src/bin/ccconnect.rs |
| 设置读取 | 读取系统配置供前端使用 | src/bin/ccsettings.rs, settings.json |
| 技能声明 | 排序能力协议声明 | skill.json |

### 4.3 已实现能力（与论文方案对齐）
1. Everything 显式命令检索与自然语言自动触发。
2. 排序技能先计划后执行，支持 pending 与 apply 流程。
3. 排序完成后输出 changelog.json 以记录变更。
4. 设置项支持工作目录、系统提示词、思考参数等运行时注入。

## 5 关键流程设计

### 5.1 检索流程
1. 用户自然语言输入检索请求。
2. 主进程解析是否命中检索意图。
3. 调用 cceverything 获取候选结果。
4. 将结果注入模型上下文，返回可读回答与路径建议。

### 5.2 整理流程
1. 用户输入整理意图或 sort 指令。
2. 系统扫描目标目录并生成计划草案。
3. 模型辅助生成整理思路与目录重命名映射。
4. 展示待执行计划并等待用户确认。
5. 用户确认后执行 apply，写入 changelog.json。

### 5.3 授权与可控流程
1. 系统通过配置目录控制可读附件范围。
2. 未授权目录默认不参与上下文读取。
3. 批量执行默认无删除动作，降低高风险变更。

## 6 实验设计与评估方法

### 6.1 评估目标
验证系统是否在“效率、可用性、安全性”三方面优于纯手工整理。

### 6.2 指标体系
1. 检索效率：从提出请求到定位目标文件的平均耗时。
2. 整理效率：一次任务中成功移动文件数与完成时间。
3. 交互负担：用户平均指令轮次。
4. 安全性：误操作率与可恢复率。
5. 资源成本：模型调用次数与估算 Token 消耗。

### 6.3 建议实验设置
1. 数据集：真实课程资料目录（文档、代码、媒体、压缩包混合）。
2. 对照组：Windows 原生搜索 + 手工整理。
3. 实验组：本系统的检索与 sort 两阶段流程。
4. 用户组：10-20 名不熟悉终端命令的大学生。

### 6.4 当前原型阶段结论
当前仓库已具备可复现实验流程，但尚未在 README 内固化统一量化数据表。建议后续在实验章节补充多轮测试统计结果，以形成完整论文中的“实验结果与讨论”部分。

## 7 可行扩展

### 7.1 引入 Embedding 混合检索
可对文件名、路径、标签与摘要进行向量化表示，形成“关键词检索 + 语义检索”混合机制。在用户仅提供模糊描述时，系统可先由 Everything 给出候选，再由向量相似度进行精排，以提升召回与精确率。

### 7.2 AI 增强终端与知识库补全
可在桌面侧集成 AI 辅助命令行，提供命令建议、参数解释与图形化导航（如目录切换可视化）。同时可沉淀本地命令知识库（如 Git 常用命令），降低新手用户对纯文本终端的学习负担。

## 8 局限性分析
1. 模型结果受提示词与上下文质量影响，复杂任务存在不稳定性。
2. 当前排序策略以规则优先，跨学科语义分类仍有提升空间。
3. 不同机器上的 Everything 配置差异可能影响开箱体验。
4. 缺少公开基准数据集，不利于跨系统横向对比。

## 9 结论
本文基于真实工程实现，提出并验证了一种本地优先的 AI 文件整理与检索方法。该方法通过“本地高速索引 + 模型语义理解 + 两阶段安全执行”实现了效率与可控性的平衡，适配不熟悉命令行和 AI Agent 的大学生用户。项目已具备从检索到整理执行的完整闭环，并为后续语义检索增强和智能终端扩展提供了稳定基础。
